# Task Protocol — Dev Studio

Единый reference по форматам данных для всех агентов системы.
Все агенты обязаны строго следовать этим форматам при чтении и записи файлов.

Продуктовая вводная и целевая модель сущностей (в т.ч. `Task`, `Execution`, FSM): [spec.md](spec.md).

---

## Сущности

### Task

Единица работы, которой оперируют агенты. Живёт в `memory/TaskBoard.md` и в `memory/TASK_MEMORY_{hex}.yml`.

**Поля:**

- `task_id` — уникальный ключ вида `{random_hex}-{name}`, где `name: [a-z0-9-]+`
- `current_execution_id` — `id` execution, который считается текущим для задачи; `null`, если execution ещё не создавался
- `started_at` — время старта задачи в формате ISO 8601 (`null`, пока ни один execution не перешёл в `in-progress`)
- `finished_at` — время завершения задачи в формате ISO 8601 (`null`, пока финальный этап не завершён успешно — см. правила ниже)
- `description` — описание задачи, поставленное пользователем

**Текущий этап SDLC** в памяти задачи **не хранится отдельным полем у Task**: его определяют по `current_execution_id` и полю `agent_type` соответствующего execution в списке `executions`.

**Типы агентов (`AgentType`)** — задаются на каждом execution, не в корне задачи:

| `agent_type` | Описание |
| --- | --- |
| `analysis` | Агент-аналитик |
| `architect` | Агент-архитектор |
| `development` | Агент-разработчик |
| `code-review` | Агент код-ревью |
| `testing` | Агент-тестировщик |
| `tech-writer` | Агент технический писатель |

**Повтор этапов в потоке:** у разных записей `executions` одной задачи значение `agent_type` может **многократно повторяться** (подряд или через другие этапы) — каждая запись это **отдельный** запуск с новым `id`. **Оркестратор** вправе поставить в очередь следующий execution с **любым** типом из таблицы выше, если это оправдано ситуацией, в том числе **снова выбрать этап**, который уже был в истории (например повторный `analysis` или второй подряд `development`). Ограничения на выбор следующего типа: после **`status: done`** — только варианты из FSM для текущего `agent_type`; после **`status: fail`** — решение по анализу всей истории, без привязки к «одному шагу FSM»; всегда — лимит **15** executions на задачу.

**FSM — допустимые переходы между этапами** при **`status: done`** у завершённого execution (оркестратор выбирает следующий `agent_type` для нового execution только в этом случае):

```text
analysis      → [architect]
architect     → [analysis, development]
development   → [architect, code-review]
code-review   → [development, testing]
testing       → [development, tech-writer]
tech-writer   → []   // конец цепочки; задача считается завершённой после execution со статусом done
```

Первый execution в задаче создаётся с `agent_type: analysis` (если иное не зафиксировано политикой проекта).

### Execution

Фиксирует один запуск агента на конкретном этапе. Живёт в `memory/TASK_MEMORY_{hex}.yml`.

Для каждого запуска агента создаётся отдельная запись `Execution`.

**Поля:**

- `id` — уникальный порядковый номер execution внутри задачи
- `parent_execution_id` — ссылка на родительский execution, из которого вырос текущий запуск; для первого execution — `null`
- `agent_type` — тип агента для этого запуска; значение из таблицы `AgentType` выше
- `started_at` — время, когда execution был переведён в `in-progress`; до старта `null`
- `finished_at` — время, когда execution был переведён в `done` или `fail`; до завершения `null`
- `output_data` — результат работы агента (требования, архитектура, код, замечания, тестовый отчёт и т.д.)
- `status` — текущий статус выполнения этого запуска (жизненный цикл execution, не этап SDLC)

**Лимит executions на задачу:** в списке `executions` допускается не более **15** записей на одну задачу (включительно). Перед созданием нового execution оркестратор проверяет, что после добавления записи лимит не будет превышен; иначе новый execution **не** создаётся — задача останавливается, пользователю сообщается о достижении потолка.

**Маршрутизация при `fail`:** если текущий execution завершён со статусом `fail`, оркестратор **не** применяет к выбору следующего шага таблицу FSM «как после `done`» в автоматическом виде. Он **анализирует всю** цепочку `executions` (типы этапов, статусы, содержимое `output_data` провалившегося и при необходимости предыдущих шагов) и **самостоятельно** решает, какой `agent_type` у следующего execution: например, снова `analysis` при противоречиях в требованиях, `architect` при необходимости пересмотра решения, `development` при ошибке реализации, повтор того же этапа, если это уместно по причине сбоя, и т.д. Новый execution создаётся с выбранным `agent_type`, `status: new`, пустыми метками времени и `output_data: ""`; рекомендуется выставить `parent_execution_id` равным `id` провалившегося execution для прослеживаемости. Затем обновляется `current_execution_id` и запускается соответствующий субагент.

> **Важно:** этап пайплайна кодируется только в `agent_type`. Поле `status` отвечает за то, создан ли execution, идёт ли работа и успешно ли она завершилась. Следующий `agent_type` после **`done`** выбирает **оркестратор** по FSM (внутри которого уже заложены возвраты и «неуникальный» по времени порядок этапов); после **`fail`** — по анализу истории и правилу лимита executions (см. выше). Повтор одного и того же этапа в очереди — **нормальная** практика, если оркестратор так решил.
> **Входные данные агента** — агент читает `output_data` предыдущего завершённого execution напрямую из `memory/TASK_MEMORY_{hex}.yml`. Поле `input_data` отсутствует во избежание дублирования данных.

**Статусы `Execution` (линейный переход):**

```text
new → in-progress → done
                 └→ fail
```

| Status | Описание |
| --- | --- |
| `new` | Execution создан оркестратором; субагент ещё не зафиксировал старт этапа |
| `in-progress` | Субагент зафиксировал старт в памяти и выполняет этап |
| `done` | Агент завершил работу успешно |
| `fail` | Агент завершил работу с ошибкой |

**Инвариант порядка (`new` → `in-progress` → работа):** пока запись execution в YAML имеет `status: new`, субагент **не** выполняет содержательную работу по задаче (рассуждения по требованиям, проектирование, правки кода, ревью, составление тест-плана, генерация документов в `docs/`, запуск команд ради результата этапа, заполнение итогового `output_data` и т.п.). Разрешено только минимально необходимое чтение `memory/TASK_MEMORY_{hex}.yml` и `memory/TaskBoard.md` для проверки корректности подготовленного execution и, при нарушении протокола, перевод в `fail` с пояснением в `output_data`. Сразу после успешной проверки субагент **обязан** записать в память переход `new` → `in-progress`, заполнить `started_at` у execution (и при первом этапе — `started_at` у задачи), **сохранить файл**, и только затем выполнять этап.

## Форматы файлов

Доска задач и файлы памяти (`TASK_MEMORY_*.yml`) лежат в каталоге **`memory/`** в корне репозитория. Пути ниже — с префиксом `memory/`.

### TaskBoard

Файл: `memory/TaskBoard.md`. Список всех активных задач. Оркестратор читает этот файл для выбора следующей задачи.

Поле **`State`** на доске — не дублирует структуру `Task` в YAML: это **человекочитаемая метка этапа**, которую поддерживает оркестратор в согласовании с файлом памяти `memory/TASK_MEMORY_*.yml` (текущий или последний завершённый `agent_type`, либо `new` для задачи без файла памяти / без executions).

```markdown
# TaskBoard

## a3f2b1-auth-feature
State: new
Description: Добавить JWT-авторизацию в API

## 9c1d4e-user-profile
State: analysis
Description: Профиль пользователя с аватаром и настройками
```

**Правила:**

- Каждая задача — отдельный `## {task_id}` раздел
- `State: new` — задача ещё не взята в работу: нет файла `memory/TASK_MEMORY_{hex}.yml` или в нём пустой список `executions` (оркестратор ещё не создал первый execution)
- После завершения пайплайна задача удаляется из TaskBoard или помечается `State: done` и больше не берётся в работу
- Оркестратор берёт задачи в порядке их появления
- Обновление `State` на доске после шага агента выполняет **оркестратор** (при планировании следующего execution или после перечитывания памяти), чтобы не расходились доска и `agent_type` в памяти

### TASK_MEMORY

Файл: `memory/TASK_MEMORY_{hex}.yml` — память конкретной задачи в формате **YAML**. Создаётся оркестратором при первом взятии задачи в работу, до запуска первого субагента.
`{hex}` — первая часть `task_id` до дефиса.

Пример для задачи `a3f2b1-auth-feature` → файл `memory/TASK_MEMORY_a3f2b1.yml`:

```yaml
task_id: a3f2b1-auth-feature
current_execution_id: 3
started_at: "2026-04-16T10:00:00Z"
finished_at: null

executions:
  - id: 1
    parent_execution_id: null
    agent_type: analysis
    started_at: "2026-04-16T10:00:00Z"
    finished_at: "2026-04-16T10:05:00Z"
    output_data: |
      ## Функциональные требования
      - POST /auth/login → возвращает access_token и refresh_token
      - POST /auth/refresh → обновляет access_token
    status: done

  - id: 2
    parent_execution_id: 1
    agent_type: architect
    started_at: "2026-04-16T10:05:30Z"
    finished_at: "2026-04-16T10:12:00Z"
    output_data: |
      ## Обзор решения
      ...
    status: done

  - id: 3
    parent_execution_id: 2
    agent_type: development
    started_at: "2026-04-16T10:12:15Z"
    finished_at: null
    output_data: ""
    status: in-progress
```

**Правила:**

- В корне YAML **нет** поля `state` у задачи — этап определяется по execution с `id == current_execution_id` и его `agent_type`
- `current_execution_id` в корне YAML указывает на текущий execution, с которым сейчас связана задача
- Новый `Execution` добавляется в конец списка `executions` оркестратором до запуска субагента
- `parent_execution_id` указывает на предыдущий релевантный execution; в линейном успешном проходе часто это id предыдущего по времени execution; после маршрутизации из `fail` оркестратор может указать на провалившийся execution (см. раздел Execution)
- Агент читает `output_data` предыдущего завершённого execution напрямую из файла
- Первый агент (`analysis`) берёт входные данные из `Description` в `memory/TaskBoard.md`
- Оркестратор ориентируется не на последний элемент массива, а на execution с `id = current_execution_id`
- Оркестратор **обязан** создать `memory/TASK_MEMORY_{hex}.yml`, если файла ещё нет
- Оркестратор **обязан** создать новый `Execution` в начале каждого этапа и обновить `current_execution_id` в корне YAML до запуска субагента, **не превышая** лимит **15** executions на задачу (см. раздел Execution)
- Агент **обязан** использовать execution, уже созданный оркестратором для его этапа
- Агент **обязан** последовательно обновлять `status`: сначала `new` → `in-progress` (**до** любой содержательной работы по задаче и до заполнения итогового `output_data`), затем `in-progress` → `done`/`fail`
- `started_at` у execution заполняется при переводе в `in-progress`
- `finished_at` у execution заполняется при переводе в `done` или `fail`
- `started_at` у Task заполняется при переводе **первого** execution в `in-progress`
- `finished_at` у Task заполняется, когда execution с `agent_type: tech-writer` завершён со статусом `done` (успешное окончание пайплайна)
- Субагент **не** выставляет «состояние задачи» в корне YAML — только поля своего execution; следующий `agent_type` и новый execution создаёт оркестратор

## Протокол работы оркестратора

Перед каждым запуском субагента оркестратор выполняет следующую последовательность:

1. **Получить `task_id`** из `memory/TaskBoard.md`
2. **Проверить наличие `memory/TASK_MEMORY_{hex}.yml`**:
   - если файла нет — создать его с корневыми полями `task_id`, `current_execution_id: null`, `started_at: null`, `finished_at: null`, `executions: []`
3. **Прочитать `memory/TASK_MEMORY_{hex}.yml`** и проверить текущий execution, если он есть:
   - если `current_execution_id` отсутствует или равен `null` — задача ещё не запускалась, нужно создать первый execution с `agent_type: analysis` (или согласованный стартовый тип) и затем запускать субагента
   - если текущий execution в `status: in-progress` — не запускать нового субагента
   - если текущий execution в `status: new` — не запускать нового субагента: execution уже создан оркестратором для этого этапа, но агент ещё не перевёл его в `in-progress`
   - если текущий execution в `status: done` — вычислить следующий шаг по FSM и `agent_type`; при успешном завершении цепочки после `tech-writer` — выставить `finished_at` у задачи, обновить TaskBoard (`State: done` или снять задачу)
   - если текущий execution в `status: fail` — оркестратор анализирует все `executions` и решает, какой следующий `agent_type` (см. раздел Execution); если executions уже **15** — новый execution не создавать, остановиться и сообщить пользователю
4. **Определить этап запуска** и выбрать нужного субагента: после `done` — строго по FSM; после `fail` — по результатам анализа истории (не по таблице FSM для `done`)
5. **Создать новый execution** для выбранного этапа: `agent_type`, `status: new`, `started_at: null`, `finished_at: null`, `output_data: ""`, при необходимости `parent_execution_id` (см. раздел Execution), обновить `current_execution_id` — **только если** в списке executions сейчас **меньше 15** записей
6. **Запустить субагента**, передав `task_id` и `execution_id`. Субагент **не** начинает содержательную работу, пока сам не перевёл свой execution из `new` в `in-progress` и не зафиксировал это в YAML (см. инвариант в разделе Execution и протокол субагента ниже).
7. **После завершения субагента** перечитать `memory/TASK_MEMORY_{hex}.yml`, найти `current_execution_id`, проверить статус текущего execution: при `done` — допустимость следующего перехода по FSM; при `fail` — анализ истории и лимит 15 executions (см. раздел Execution)
8. **Синхронизировать `memory/TaskBoard.md`**: выставить `State` в соответствии с текущим или только что запланированным этапом (см. раздел TaskBoard)

## Протокол работы агента (субагента)

Каждый агент-субагент при запуске выполняет следующую последовательность:

1. **Получить `task_id` и `execution_id`** из prompt оркестратора
2. **Прочитать `memory/TASK_MEMORY_{hex}.yml`** — найти запись execution, подготовленную оркестратором для этого запуска (по `execution_id` и `current_execution_id`). На этом шаге **не** выполняйте содержательную работу по задаче: только проверка структуры и полей, необходимых для шага 3
3. **Проверить подготовленный execution**:
   - `id` должен совпадать с `execution_id`
   - `current_execution_id` в корне YAML должен совпадать с `execution_id`
   - `agent_type` должен совпадать с ролью текущего агента
   - `status` должен быть `new`
4. **Если проверка не пройдена** — зафиксируйте нарушение в `output_data`, выставьте `status: fail`, заполните `finished_at` и **остановитесь** (без перевода в `in-progress`)
5. **Зафиксировать старт этапа в памяти**: обновить execution → `status: in-progress`, заполнить `started_at` у execution; **если `Task.started_at` ещё пустой** — заполнить его текущим временем. Сохранить `memory/TASK_MEMORY_{hex}.yml`. **Содержательная работа по задаче разрешена только после этого шага**
6. **Прочитать входные данные**:
   - при необходимости перечитать `memory/TASK_MEMORY_{hex}.yml` целиком и историю `executions`
   - **Прочитать `memory/TaskBoard.md`** — убедиться, что описание задачи на месте (сверка `task_id`); поле `State` на доске обновляет оркестратор, не субагент
   - взять `output_data` последнего завершённого execution (`status: done` или последнего релевантного по политике чтения цепочки)
   - либо взять `Description` из `memory/TaskBoard.md`, если завершённых executions ещё нет
7. **Выполнить работу** согласно своей роли (код, ревью, тест-план, документы в `docs/` и т.д.)
8. **Записать результат**: заполнить `output_data`, обновить `status: done` или `status: fail`, заполнить `finished_at`, сохранить YAML
9. **Не** добавлять в корень YAML поле состояния задачи; **не** менять `agent_type` у уже созданного execution после старта этапа

После этого субагент завершает работу. Оркестратор по завершённому execution, его `agent_type`, `status`, FSM (для `done`) и анализу истории (для `fail`) решает, создавать ли следующий execution и как обновить TaskBoard / `finished_at`, с учётом лимита **15** executions.
